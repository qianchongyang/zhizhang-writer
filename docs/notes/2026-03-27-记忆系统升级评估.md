# 记忆系统升级评估 (v5.9.0)

> 评估时间：2026-03-27

## 新增功能总结

| 功能 | 说明 | 评估 |
|------|------|------|
| **story_memory.json** | 跨章节稳定记忆层，替代分散的 state.json | 核心架构正确 |
| **structured_change_ledger** | 通用结构化变化记录，替代 numeric_ledger | 扩展性强 |
| **change_kind 推断** | 基于关键词自动推断变化类型 | 实用，减少人工 |
| **memory_score / memory_tier** | 重要性评分 + 三层分级 | 分层清晰 |
| **容量限制** | character(≤10) / events(≤50) / snapshots(≤20) / ledger(≤50) | 防膨胀 |
| **每5章自动触发** | consolidation 里程碑 + last_consolidated_chapter | 自动整理 |

## 与原设计文档对比

| 设计点 | 预期 | 实际 | 符合度 |
|--------|------|------|--------|

## 评估结论

### 优点
1. 架构完整，设计文档到代码实现一致性高
2. change_kind 推断实用，减少人工标记负担
3. 容量限制 + 自动淘汰，防止数据膨胀
4. 伏笔同步机制（plot_threads 直接从 state 同步）
5. 旧项目可直接升级，无需重建底座

### 待改进/待实现
1. **召回接口缺失** - story_memory 写入了，但 Context Agent 写前召回逻辑未接入
2. memory_score 阈值硬编码，无法运行时调整
3. 缺乏针对 `memory_tier` 的差异化召回策略

## PostgreSQL 评估

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| **当前 JSON 文件** | 零依赖、原子写入、简单 | 检索能力弱、无事务 | 50章内、变更不频繁 |
| **SQLite** | 结构化、事务、轻量 | 向量能力需扩展 | 中等规模（50-200章） |
| **PostgreSQL+pgvector** | ACID、向量检索、复杂查询 | 依赖重、迁移复杂 | 200章+、高并发 |

**建议路径**：先验证当前文件方案在 50 章后的召回效果，如果向量检索成为瓶颈再迁移到 PostgreSQL。

## v5.10.0 详细设计思路

**目标**：让 Context Agent 在写前能主动召回 story_memory

**最小改动点**：
1. `context_agent.md` - 新增"从 story_memory 加载记忆上下文"步骤
2. 导出 `story_memory` 中的 `characters.current_state` 和 `recent_events` 给 LLM
3. 召回触发条件：`chapter > 10` 时启用（10章内用前3章摘要足够）

**测试验证**：
- 写第50章，检查召回的角色状态是否与第45章一致
- 检查 recall 精度（召回了多少是相关的）
