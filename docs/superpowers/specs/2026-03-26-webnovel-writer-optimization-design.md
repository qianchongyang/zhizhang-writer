# Webnovel Writer 二次开发优化设计方案

## 项目背景

### 现状问题

1. **Token消耗过高**：每章100-150次API调用，MiniMax CODING PLAN每5小时1500次额度，约5分钟消耗完毕
2. **上下文累积**：长篇小说写作时，上下文窗口被污染，约2章后需清空重开
3. **无法自动化**：完成1章后需手动触发下一章，无法持续创作

### 优化目标

| 目标 | 当前 | 目标 |
|------|------|------|
| 每章API调用 | 100-150次 | 50次以内 |
| 主窗口上下文 | 全量累积 | 精简70% |
| 连续写作能力 | 2章需清空 | 不限 |
| 自动化支持 | 无 | 批量+夜间模式 |

---

## 实施顺序

**B → A → C**（技术依赖关系决定）

```
B (Agent Teams架构)  ← 最先，基础设施
       ↓

       ↓
```

## 方案1：混合架构设计（推荐）

### 整体架构

```
┌─────────────────────────────────────────────────────────────────┐
│                    主Agent（写作核心 + 编排）                     │
│  ┌─────────────┐                                                │
│  │ 写作上下文  │ ← 仅保留：当前章大纲 + 必要设定 + 写作风格指南   │
│  └─────────────┘                                                │
└─────────────────────────────────────────────────────────────────┘
         │                    │                    │
         ▼                    ▼                    ▼
┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐
│  Context Agent  │  │    审查器池     │  │   Data Agent   │
│   (独立窗口)     │  │   (分组并行)    │  │   (独立窗口)    │
│                 │  │                 │  │                │
│ 输入：项目根目录 │  │ 组1：一致性     │  │ 输入：章节文件  │
│ 输出：精简任务书 │  │       连贯性    │  │ 输出：状态更新  │
│                 │  │       OOC       │  │                 │
│ Token节省：40%  │  │                 │  │ Token节省：30%  │
│                 │  │ 组2：追读力     │  │                 │
│                 │  │       爽点      │  │                 │
│                 │  │       节奏     │  │                 │
└─────────────────┘  └─────────────────┘  └─────────────────┘
         │                    │                    │
         ▼                    ▼                    ▼
   【中间状态文件：.webnovel/tmp/agent_outputs/】
```

### 核心设计原则

1. **主Agent最小化上下文** - 只保留当前章必需信息
2. **Agent间通过文件交换** - 不依赖内存传递
3. **分组并行审查** - 减少窗口数量，同时保持隔离
4. **结果摘要化** - 每个Agent只返回关键结论，不返回完整处理过程

## 方案2：数据流设计

### 完整数据流（写一章）

```
1. 主Agent → Context Agent（独立窗口）
   ┌──────────────────────────────────────┐
   │ 输入：                               │
   │   - project_root                    │
   │   - state.json（进度/strand/伏笔）  │
   │   - index.db（核心实体/关系）       │
   │   - 章节大纲                        │
   │   - 上章摘要                        │
   │ 输出（写入文件）：                   │
   │   - .webnovel/tmp/ctx_ch{N}.json   │
   └──────────────────────────────────────┘

2. 主Agent ← Context Agent（读文件）
   主Agent仅加载：task_summary + constraints + style_guide

3. 主Agent → 审查器组1（独立窗口）
   ┌──────────────────────────────────────┐
   │ 并行执行：一致性 + 连贯性 + OOC       │
   │ 输出（写入文件）：                     │
   │   - .webnovel/tmp/rev1_ch{N}.json    │
   └──────────────────────────────────────┘

4. 主Agent → 审查器组2（独立窗口）
   ┌──────────────────────────────────────┐
   │ 并行执行：追读力 + 爽点 + 节奏        │
   │ 输出（写入文件）：                     │
   │   - .webnovel/tmp/rev2_ch{N}.json    │
   └──────────────────────────────────────┘

5. 主Agent → Data Agent（独立窗口）
   ┌──────────────────────────────────────┐
   │ 输入：章节文件 + review_score         │
   │ 输出（写入文件）：                     │
   │   - .webnovel/tmp/data_ch{N}.json    │
   └──────────────────────────────────────┘

6. 主Agent（最终润色 + 合并结果）
```

### 中间文件结构

```json
// .webnovel/tmp/agent_outputs/ctx_ch0100.json
{
  "chapter": 100,
  "timestamp": "2026-03-26T10:00:00",
  "task_summary": "萧炎在天云宗获得传承，突破斗师",
  "constraints": ["主线不动摇", "战力严格递增"],
  "style_guide": "战斗场面详尽，感情线克制",
  "entities_in_chapter": ["xiaoyan", "master", "tianyun_sect"]
}
```

### 数据流的关键改进点

| 当前设计 | 改进后 | 节省 |
|----------|--------|------|
| 主Agent加载全部上下文 | 只加载task_summary | ~30% token |
| 审查器串行执行 | 组内并行 | ~50% 时间 |
| 各Agent重复读取state/index | 通过文件传递增量 | ~20% API调用 |

## 方案3：分阶段实施路径

### 阶段1：Agent Teams架构（预计2-3周）

**目标**：实现混合架构，上下文隔离

| 步骤 | 任务 | 产出 |
|------|------|------|
| 1.1 | 设计中间层协议 | `docs/agent-protocol.md` |
| 1.2 | 改造Context Agent独立输出 | `context_agent_result.json` |
| 1.3 | 实现审查器分组并行 | `review_group{N}_result.json` |
| 1.4 | 改造Data Agent独立输出 | `data_agent_result.json` |
| 1.5 | 主Agent精简上下文加载 | 仅加载摘要 |
| 1.6 | 端到端测试 | 写完一整章 |

**验收标准**：
- 主窗口上下文大小降低50%
- 每章总API调用次数降到80次以内

---

### 阶段2：Token优化（预计1-2周）

**目标**：在阶段1基础上进一步降低调用

| 优化点 | 当前 | 优化后 | 方法 |
|--------|------|--------|------|
| Context Agent | 15-20次 | 8-10次 | 批量SQL查询 |
| 审查器组 | 60-90次 | 20-30次 | 分组并行 + 精简输出 |
| Data Agent | 15-20次 | 8-10次 | 批量写入 |
| 润色 | 多次 | 1-2次 | 汇总修复 |

**验收标准**：
- 每章API调用降到 **50次以内**
- 预计节省60-70% Token

---

### 阶段3：自动化创作（预计1-2周）

**目标**：持续自动写作，直到完成或手动停止

| 功能 | 实现方式 |
|------|----------|
| 批量写入 | `webnovel-write --batch --from 101 --to 200` |
| 进度跟踪 | `.webnovel/workflow_batch.json` |
| 异常处理 | 断点自动记录 + `webnovel-resume` |
| 夜间模式 | `--night-mode --max-calls 1400`（留100次余量）|

**验收标准**：
- 可一次性写完一本书（假设800章）
- 期间无需人工干预
- 异常自动处理，不丢失进度

## 方案4：风险与应对

### 主要风险

| 风险 | 等级 | 应对策略 |
|------|------|----------|
| **Agent间状态不一致** | 🔴 高 | 中间文件版本号 + checksum校验 |
| **审查结果丢失** | 🔴 高 | 写入前先写.tmp，完成后rename |
| **上下文隔离不彻底** | 🟡 中 | 每个Agent独立工作目录 |
| **批量写作质量下降** | 🟡 中 | 设置质量阈值，低于阈值自动停下 |
| **异常恢复复杂** | 🟡 中 | 详细的断点日志 + resume支持 |

### 状态持久化协议

```
每次Agent执行前：
1. 读取状态文件
2. 检查版本号
3. 执行操作
4. 写入临时文件（.tmp）
5. rename到正式位置

异常恢复时：
- 读取最近的.tmp文件
- 检查checksum
- 决定是重做还是继续
```

### 回滚机制

| 场景 | 回滚方式 |
|------|----------|
| 单章失败 | 删除本章文件，重置chapter_meta |
| 批量中断 | 从上一个成功章节继续 |
| 数据损坏 | 从git历史恢复 |
