# Dynamic Outline Adjustment

Purpose: 动态调整已有大纲，解决冲突、插入副本、增强章节。

## Project Root Guard
- 必须先解析 `PROJECT_ROOT` 为真实书项目根（必须包含 `.webnovel/state.json`）

环境设置：
```bash
export WORKSPACE_ROOT="${CLAUDE_PROJECT_DIR:-$PWD}"

if [ -z "${CLAUDE_PLUGIN_ROOT}" ] || [ ! -d "${CLAUDE_PLUGIN_ROOT}/skills/webnovel-adjust" ]; then
  echo "ERROR: 未设置 CLAUDE_PLUGIN_ROOT" >&2
  exit 1
fi
export SKILL_ROOT="${CLAUDE_PLUGIN_ROOT}/skills/webnovel-adjust"
export SCRIPTS_DIR="${CLAUDE_PLUGIN_ROOT}/scripts"

export PROJECT_ROOT="$(python "${SCRIPTS_DIR}/webnovel.py" --project-root "${WORKSPACE_ROOT}" where)"
```

## Adjustment Types

### Type 1: 小修 - 修改某章情节
适用场景：某章的大纲内容需要修改（如筑基丹被偷 → 灵石被盗）

```bash
# 读取当前大纲
cat "$PROJECT_ROOT/大纲/第{volume_id}卷-详细大纲.md"
cat "$PROJECT_ROOT/大纲/第{volume_id}卷-时间线.md"
```

### Type 2: 中修 - 插入新章节/副本
适用场景：在指定位置插入新情节（如宗门大比10章副本）

```bash
# 插入后需要重新编号
# 所有后续章节的编号自动 +N
```

### Type 3: 大修 - 重写卷大纲
适用场景：整卷的核心冲突需要调整

## Workflow

### Step 1: 确认调整范围
```
用户：请调整第 {N} 章的大纲
原因：{用户描述的问题}

或：

用户：在第 {N} 章后插入 {M} 章的 {副本名} 副本
```

### Step 2: 分析影响
1. 读取当前大纲和时间线
2. 分析调整对后续章节的影响
3. 生成影响报告

**影响报告格式**：
```
调整范围：第 {N} 章
影响：
- 后续 {M} 章需要重新编号
- 时间线需要顺延 {X} 天
- 倒计时事件：宗门大比 D-{old} → D-{new}
```

### Step 3: 确认调整方案
展示调整后的：
- 章节大纲变化
- 时间线变化
- 需要更新的相关章节

### Step 4: 执行调整
1. 修改大纲文件
2. 更新时间线
3. 同步 state.json（如有变更）
4. 记录调整历史

### Step 5: 验证
- 检查调整后的大纲一致性
- 检查时间线逻辑
- 检查倒计时是否正确

## 副本插入模板

当用户要求插入副本时，使用以下模板：

```markdown
## {副本名称}

**类型**: {战斗副本/情感副本/探秘副本}
**起始章节**: 第 {N} 章
**章节数**: {M} 章
**核心冲突**: {一句话描述}
**起始时间**: {时间锚点}
**结束时间**: {时间锚点}
**与主线关系**: {如何影响主线}

### 副本节拍
| 阶段 | 章节 | 内容 |
|------|------|------|
| 开局 | N~N+2 | 副本触发 |
| 发展 | N+3~N+6 | 困难推进 |
| 高潮 | N+7~N+8 | 最终对抗 |
| 收尾 | N+9~N+10 | 回归主线 |

### 副本内关键实体
- 新角色：
- 新物品：
- 新地点：
```

## 铺垫章节自动插入

当审查阶段检测到"关系跳跃"或"节奏太快"时，自动生成铺垫章节：

### 铺垫章节生成规则

| 问题类型 | 插入位置 | 章节数 | 内容要求 |
|---------|---------|-------|---------|
| 关系跳跃 | 当前章节前 | 3-5章 | 关系递进描写，至少3个关键事件 |
| 巧合过多 | 当前章节前 | 2-3章 | 补充道具/人物出场，解释"刚好" |
| 节奏太快 | 当前章节前 | 5-10章 | 拆分为小副本，起承转合完整 |

### 执行步骤

1. **读取当前大纲**：确定插入位置和影响范围
2. **生成铺垫章节**：按照模板生成章节大纲
3. **章节重新编号**：后续所有章节编号自动 +N
4. **更新时间线**：时间轴顺延
5. **更新节拍表**：插入新的节拍点
6. **验证一致性**：确保调整后无逻辑冲突

### 铺垫章节模板

```markdown
### 第 {N} 章：{标题}
- 目标: {20字以内}
- 阻力: {20字以内}
- 代价: {20字以内}
- 时间锚点: {相对时间，如"关系事件前3天"}
- 章内时间跨度: {如 "2小时"}
- 与上章时间差: {如 "紧接"}
- 爽点: {类型} - {30字以内}
- Strand: {Quest|Fire|Constellation}
- 反派层级: {无/小/中/大}
- 视角/主角: {主角A/女主/群像}
- 关键实体: {新增或重要出场}
- 本章变化: {30字以内}
- 章末未闭合问题: {30字以内}
- 钩子: {类型} - {30字以内}

**本章核心内容**：
{详细描述本章需要完成的关系铺垫/事件铺垫}
```

## 冲突检测规则

在调整时自动检测以下冲突：

| 冲突类型 | 检测逻辑 |
|---------|---------|
| 时间逆流 | 新时间 < 上一章时间 且 未标注闪回 |
| 物品凭空出现 | 物品在后续章节才首次出现但被使用 |
| 境界矛盾 | 境界变化不符合修炼逻辑 |
| 倒计时跳跃 | 倒计时非连续递减 |

## 大纲版本历史

每次调整后，自动记录：

```bash
# 备份当前大纲
cp "$PROJECT_ROOT/大纲/第{volume_id}卷-详细大纲.md" \
   "$PROJECT_ROOT/.webnovel/outline_history/第{volume_id}卷-详细大纲.v{version}.md"
```

## 示例对话

**用户**: 第13章的大纲有问题，"筑基丹被偷"应该是"灵石被盗"

**Claude**:
```
明白，让我先读取第13章的当前大纲：
[读取大纲]

影响分析：
- 第13章内容修改
- 不影响后续章节编号
- 不影响时间线
- 确认是否同步修改 state.json 中的 item_states

调整确认：
- 第13章：灵石被盗（实验材料）代替筑基丹被偷
- 原因：练气三层无法持有筑基丹

执行调整？ (y/n)
```

**用户**: y

**Claude**:
```
✓ 已修改：第13章大纲
✓ 已更新时间线
✓ 已记录调整历史

后续章节无需调整。
```
