# 记忆系统升级深度评估

> 评估日期：2026-03-27
> 评估范围：当前代码实现、记忆链路、召回机制、数据层边界、实施版需求与 v5.9.0 plan

## 1. 当前系统真实结构

现有项目并不是“没有记忆系统”，而是已经形成了一个偏完整但分散的记忆栈：

```text
写前上下文: ContextManager / extract_chapter_context.py
运行态状态: state.json + StateManager
结构化历史: index.db + SQLStateManager + IndexManager
语义召回: vectors.db + RAGAdapter
章节快照: snapshot_manager
学习型模式: project_memory.json + /webnovel-learn
```

### 已存在的能力

- `state.json`
  - 保存当前进度、主角状态、关系、世界设定、`plot_threads`、`chapter_meta`、`character_states`、`item_states`、`time_states`
- `index.db`
  - 已有实体、别名、状态变化、关系、关系事件、无效事实、审查指标、追读力、写作清单评分
- `vectors.db`
  - 已有向量检索和 BM25 检索
- `context_manager`
  - 已经把 state、近期摘要、章节 meta、近出场实体、reader signal、genre profile、writing guidance、`project_memory.json` 组装进写前上下文
- `status_reporter`
  - 已能从 `plot_threads.foreshadowing` 计算伏笔紧急度与超期状态

### 现有系统的问题不是“没有数据”，而是：

- 记忆没有统一分层定义
- 写入与召回契约不够明确
- 多个来源对同一事实缺少优先级
- 写前上下文复用了快照，但快照失效条件不够细
- 结构化事实、推断事实、阶段摘要混在一起，容易脏写

## 2. 代码层面的核心判断

### 2.1 `state.json` 已经承担了“当前权威状态”

`StateManager.process_chapter_result()` 已把章节处理结果落到：

- `chapter_meta`
- `character_states`
- `item_states`
- `time_states`
- `plot_threads`
- `progress`

这意味着当前系统已经在做“基础记忆写入”，并不是从零开始。

### 2.2 `index.db` 已经承担了“可查询历史”

`IndexManager` 和 `SQLStateManager` 已经把下列内容结构化：

- 实体
- 别名
- 状态变化
- 关系
- 关系事件
- 章节阅读力
- 审查指标
- 追读力债务

这说明后续所谓“记忆升级”更像是：

- 选择哪些信息该提升为长期记忆
- 哪些信息要写入故事记忆层
- 写前上下文怎么把这些信息喂给 LLM

而不是重建底层数据库。

### 2.3 `context_manager` 的瓶颈是“拼装层”，不是“数据源缺失”

`ContextManager.build_context()` 已有：

- snapshot 优先命中
- fallback 重新组装
- `core / scene / global / reader_signal / genre_profile / writing_guidance / story_skeleton / memory / preferences / alerts`

但目前 `memory` 只接了 `project_memory.json`，没有独立的故事记忆层。

这意味着当前写前召回问题不是“没地方放”，而是：

- 没有专门为故事级记忆准备的结构
- 没有明确的注入顺序
- 没有把“伏笔、里程碑、最近重大事件、章节结尾状态”统一压缩

### 2.4 快照是隐性风险点

`build_context()` 会优先读取 `snapshot_manager` 的快照。

如果后续加入 `story_memory.json`，但快照兼容条件仍然只看模板，那么会出现：

- 故事记忆已经更新
- 但写前仍复用旧 snapshot
- 结果 LLM 看到旧召回内容

所以后续方案必须加入至少一个失效因子：

- `story_memory` 版本号
- `last_consolidated_chapter`
- 或 `memory_signature`

否则“升级了记忆层”却不一定能真正进入上下文。

## 3. 当前召回机制的真实行为

### 3.1 写前召回现在是三段式

1. `state.json` 直接摘要
2. 最近章节摘要 / chapter meta / 最近出场实体
3. RAG 查询正文片段

这套机制对“最近发生了什么”有效，对“长期链路”不够稳定。

### 3.2 现有召回更擅长“像不像”，不擅长“应该记什么”

`RAGAdapter.search()` 现在支持：

- vector
- bm25
- backtrack
- graph_hybrid
- hybrid

但这些都属于文本相似性或图式检索，无法自动保证：

- 伏笔是否已回收
- 数值是否精确
- 角色里程碑是否完整
- 上章结尾状态是否入库

因此 `story_memory.json` 的价值不是替代 RAG，而是给 RAG 提供更高精度的结构化前置信息。

## 4. 这次升级真正应该解决的东西

### 必须解决

- 写前召回未回收伏笔
- 写前召回最近重大事件
- 写前召回上章结尾状态
- 数值/关系/时间变化精确落盘
- 记忆整理可重建
- 结构化记忆具备健康检查

### 不该在 v5.9.0 硬做的东西

- PostgreSQL 迁移
- 图数据库独立落地
- 跨项目共享记忆
- 复杂的语义推理引擎

这些不是没有价值，而是当前阶段会吞掉太多工程预算。

## 5. 对实施版需求的修正建议

### 5.1 `story_memory.json` 的位置要明确

建议固定为：

```text
PROJECT_ROOT/.webnovel/memory/story_memory.json
```

不要放在项目根目录，也不要和 `project_memory.json` 混用。

### 5.2 `project_memory.json` 继续保留

它更像“会话模式记忆”：

- 钩子
- 节奏
- 对话
- 微兑现

而 `story_memory.json` 是“故事事实记忆”：

- 角色阶段
- 伏笔生命周期
- 重大事件
- 数值变化
- 章节结尾状态

两者用途不同，最好不要合并。

### 5.3 必须加入记忆 ID

建议在实施版里明确：

- `foreshadowing_id`
- `event_id`
- `milestone_id`
- `chapter_snapshot_id`

否则后面更新旧记录只能靠文本匹配，维护成本会越来越高。

### 5.4 必须把“推断事实”单独标记

建议给弱记忆加：

- `confidence`
- `evidence`
- `source_chapters`

这样可以区分：

- 强事实：数值、状态、已确认关系
- 弱事实：情绪、因果、氛围

## 6. 对 v5.9.0 plan 的修正建议

### Task 2 需要补一项

新增 `snapshot_manager` / `ContextManager` 的版本失效条件：

- 当 `story_memory.json` 版本或整理章节变化时，旧 snapshot 不能直接复用

### Task 3 需要更明确

“写前召回”应明确按优先级注入：

1. `story_memory.json`
2. `state.json`
3. `index.db` 结构化历史
4. `vectors.db` 语义召回

### Task 4 需要定义整理输入

每 5 章整理不是“随便压缩”，而是只整理这些高价值项：

- 角色里程碑
- pending 伏笔
- 最近重大事件
- 章节结尾状态

### Task 6 需要健康指标落地

健康检查不要只输出“有无数据”，要给出至少：

- 命中率
- 漏召回样本
- 脏记忆样本
- 容量趋势
- 重建耗时

## 7. 结论

这次升级可以做，但正确的实现路径不是“重构底层存储”，而是：

1. 先固化记忆分层和写入契约
2. 再新增 `.webnovel/memory/story_memory.json`
3. 再把它接入写前上下文
4. 再补周期整理、健康检查、回退重建
5. 最后再考虑更重的数据库升级

如果按这个顺序做，风险最低，也最符合当前仓库的真实状态。
