# 记忆系统升级需求评估

> 评估时间：2026-03-27
> 评估对象：`docs/notes/2026-03-27-记忆系统升级需求.md`

## 结论

这份需求判断方向基本正确，但当前版本有两个明显问题：

1. **问题定义是对的，解法有些过重**
2. **缺少“记忆写入规范”和“正确性校验标准”**

更准确地说，当前项目不是“完全没有记忆系统”，而是：

- 已有 `state.json` 承担运行态记忆
- 已有 `index.db` 承担实体/关系/状态变化索引
- 已有 `vectors.db` 承担语义召回
- 但**没有统一的记忆分层定义、准入规则、召回编排、健康检查**

所以这个需求应该收敛为：

> 在现有三层数据层之上，补一层“可被 LLM 稳定消费的故事记忆层”，而不是立刻整体迁移到 PostgreSQL + 图数据库。

## 哪些判断成立

### 1. 长程召回确实是核心痛点

这个判断成立。现有系统更擅长：

- 当前状态查询
- 最近章节语义召回
- 实体索引与别名消歧

但不擅长：

- 角色跨 30-100 章成长曲线
- 伏笔从埋设到兑现的生命周期召回
- “为什么现在必须这样写”的因果提示

### 2. 数值状态不能只靠 LLM 推断

这个判断完全成立，而且仓库里已经有这条路线的基础：

- `state.json` 已有 `item_states` / `time_states`
- `index.db` 已有 `state_changes`
- `update_state.py` 已有伏笔状态更新

真正缺的是：

- 数值字段 schema
- 单位/币种换算规则
- 更新冲突处理
- 写前校验和写后校验

### 3. 分层记忆思路是正确的

参考 Mem0 / OpenClaw 的“分层记忆”是合理的。项目里当前最适合抽成四层：

- 运行态记忆：`state.json`
- 结构化历史：`index.db`
- 语义文本记忆：`vectors.db`
- 项目级提炼记忆：新增 `story_memory.json`

这比“一把梭图数据库”更符合现阶段代码基线。

## 哪些地方过度设计

### 1. 现在就切 PostgreSQL，不合适

需求文档里把 PostgreSQL 说成主推荐方案，这在理论上没问题，但对当前仓库不经济。

原因：

- 现有实现全面围绕 `state.json + SQLite + vectors.db`
- `StateManager` 已处理锁、增量合并、SQLite 同步
- `index.db` 已经承担状态变化和查询索引
- 真正瓶颈是“记忆建模与召回策略”，不是 SQLite 本身先天不够

结论：

- **短中期不建议因为记忆升级直接切 PostgreSQL**
- 应先做“SQLite 兼容方案 + 新增 story_memory 层”
- 只有在下面条件同时成立时，再评估迁移：
  - 单项目数据量明显拖慢查询
  - 需要跨表事务的一致性问题频繁暴露
  - 需要复杂时序查询/多用户/远程服务化

### 2. 图数据库不是 P0

关系型图谱是有价值的，但当前需求最先缺的是：

- 写前召回内容质量
- 伏笔生命周期管理
- 事件摘要沉淀
- 数值状态精确更新

这些都可以先用现有 SQLite 结构和 JSON 记忆层解决。

### 3. “每 5 章整理”不能作为唯一触发条件

这个规则适合做“批量巩固”，但不适合承担全部记忆更新。

至少要拆成两类：

- 实时更新：数值变化、伏笔状态、关系变化、章节结尾状态
- 延迟整理：角色里程碑、阶段摘要、近 5 章事件压缩

否则会出现“第 46 章写前拿不到第 45 章刚发生的重要变化”的问题。

## 必须补充的内容

### 1. 记忆写入契约

现在文档回答了“存什么”，但没有把“何时写、谁来写、写成什么格式、失败怎么办”说透。

建议补一张表：

| 记忆类型 | 写入时机 | 写入者 | 存储位置 | 失败处理 |
|---------|---------|-------|---------|---------|
| 数值记忆 | 章节落盘后实时 | Data Agent | `state.json` + `index.db` | 拒绝静默失败，写日志并重试 |
| 伏笔状态 | 埋设/回收当章 | Data Agent | `plot_threads.foreshadowing` | 未命中则标红待人工确认 |
| 章节结尾状态 | 每章 | Data Agent | `chapter_meta` | 缺字段则给默认空值 |
| 阶段记忆 | 每 5 章 | Memory Consolidator | `story_memory.json` | 可重建，失败不阻塞写作 |

### 2. Source of Truth 归属

需求里没有明确“同一信息以谁为准”。

必须明确：

- 当前状态类：以 `state.json` 为准
- 历史变化类：以 `index.db.state_changes` 为准
- 语义场景类：以 `vectors.db` 为准
- 高价值摘要类：以 `story_memory.json` 为准

否则会出现：

- `story_memory.json` 说角色闭关
- `state.json` 说角色已出关
- LLM 不知道信谁

### 3. 记忆 ID 与可更新性

如果没有稳定 ID，后续“更新旧记忆”会越来越脏。

建议所有可更新对象都加 ID：

- 伏笔：`foreshadowing_id`
- 事件：`event_id`
- 角色里程碑：`milestone_id`
- 数值账户：`ledger_id`

同时定义状态机：

- `pending`
- `active`
- `fulfilled`
- `cancelled`
- `invalidated`

### 4. 记忆提取置信度

隐式状态最容易脏写。

建议新增：

- `confidence`
- `evidence_chapters`
- `evidence_excerpt` 或 scene 引用

特别是：

- 情绪时间线
- 因果语义记忆
- 关系变化

这类不要直接当强事实使用，应分成“强事实”和“推断事实”。

### 5. 遗忘策略不能只按“最旧”

文档里写“淘汰最旧记录”，这个不够。

应改成多因素淘汰：

- 已兑现伏笔优先归档
- 长期无引用且低重要度事件优先淘汰
- 主角/主线/核心设定永不自动淘汰
- 近 20 章高频引用的记忆禁止淘汰

### 6. 健康检查指标

这部分是当前需求最缺的。

建议补 5 个指标：

- 记忆命中率：写前注入的记忆有多少被实际引用
- 漏召回率：审查时发现应召回但没召回的比例
- 脏记忆率：与 `state.json`/`index.db` 冲突的比例
- 记忆膨胀率：总条目数、总字数增长趋势
- 记忆重建耗时：从原始数据重算 `story_memory.json` 的耗时

### 7. 回退与重建机制

需求里没有写“记忆坏了怎么办”。

这必须补：

- `story_memory.json` 必须是**可删除可重建**
- 提供重建命令，例如：
  - `/webnovel-memory rebuild`
  - 或统一到现有 CLI
- 重建来源：
  - `state.json`
  - `index.db`
  - 最近 N 章正文 / RAG chunks

## 建议重排优先级

当前文档的优先级方向基本对，但建议改成更可执行的版本：

### P0

- 写前自动召回未回收伏笔
- 写前自动召回最近 3-5 章重大事件
- 写前自动召回上章结尾状态

这是最小闭环，投入小，收益最大。

### P1

- 新增 `story_memory.json`
- 每 5 章做一次记忆整理
- 角色里程碑沉淀

### P2

- 结构化变化账本化
- 时间约束精确追踪
- 关系变化历史快照

### P3

- 情绪时间线
- 因果/语义记忆
- 更复杂的角色弧线分析

### Deferred

- PostgreSQL
- 图数据库
- 跨项目共享记忆

## 建议的数据落地方案

### 推荐方案

在不推翻现有架构的前提下，新增一层：

```text
state.json        -> 当前权威状态
index.db          -> 结构化历史与查询索引
vectors.db        -> 场景语义召回
story_memory.json -> 高价值提炼记忆（给 LLM 直接吃）
```

### `story_memory.json` 建议结构

比当前设计再补 4 个字段：

- `source_of_truth`
- `confidence`
- `evidence`
- `updated_by`

示例：

```json
{
  "version": "1",
  "last_consolidated_chapter": 45,
  "characters": {
    "萧炎": {
      "current_state": "斗王巅峰，闭关中",
      "source_of_truth": "state.json",
      "confidence": 1.0,
      "evidence": [{"chapter": 45, "type": "state_change"}],
      "milestones": [
        {
          "id": "ms_xiaoyan_45_breakthrough",
          "chapter": 45,
          "event": "突破斗王巅峰",
          "confidence": 0.98
        }
      ]
    }
  }
}
```

## 最关键的一条补充

这套需求还缺一个明确目标：

> 记忆系统升级最终要优化哪个可量化结果？

建议在需求顶部补 3 个验收指标：

- 写前上下文长度下降多少
- 伏笔漏回收率下降多少
- 数值/状态一致性错误下降多少

没有这个，后面很容易陷入“加了很多记忆结构，但写作质量未显著提升”。

## 最终建议

可以推进，但建议改成下面这个版本：

1. **不改数据库底座，先补 `story_memory.json` 记忆层**
2. **把“每 5 章整理”改成“实时更新 + 批量巩固”双机制**
3. **先做强事实记忆，再做情绪/因果等弱事实记忆**
4. **补齐 Source of Truth、ID、置信度、重建机制、健康指标**
5. **把 PostgreSQL / 图数据库降级为远期选项，而不是当前主方案**

这样做风险最低，也最符合当前代码基线。
