# inkos 项目评估

评估对象：`https://github.com/Narcooo/inkos`

## 一句话结论

`inkos` 的价值不在“写小说”本身，而在于它把长篇创作拆成了可编译、可审计、可回滚的流水线；对我们的项目最有启发的是“控制面 / 真相层 / 运行时产物”三分法，以及 `plan -> compose -> write -> audit -> revise` 的硬流程。

## 值得借鉴的点

1. **把创作输入拆成控制面**
   - `author_intent.md`：长期意图
   - `current_focus.md`：近期关注点
   - `chapter-XXXX.intent.md`：本章任务书
   - 这个结构比把全部 prompt 堆在一起更稳定，也更适合排查问题。

2. **把“真相”与“展示”分离**
   - 他们把 `story/state/*.json` 作为事实来源，markdown 只是人类可读投影。
   - 这和我们现在的 `state.json + story_memory + dashboard` 思路一致，说明方向是对的。

3. **长篇流程是可编排的**
   - `plan` / `compose` / `draft` / `audit` / `revise` / `write next`
   - 这比“单次生成”强很多，尤其适合我们这种要兼顾上下文、记忆、一致性和追读力的系统。

4. **明确记忆检索层**
   - 他们用 SQLite 时序记忆数据库做相关性检索。
   - 这给我们的启发是：记忆层不是“附加项”，而是写前编译链的一部分。

5. **审计驱动修订**
   - 不是写完就结束，而是写完必须过审计，再决定是否修订。
   - 这很适合我们继续加强 `Consistency / Reader-pull / OOC` 的闭环。

## 不建议直接照搬的点

1. **10-agent 可能过重**
   - 适合做大型工程化写作流水线，但对我们当前阶段可能过度复杂。
   - 我们更适合先把 2 个核心 Agent 和 1 套稳定运行时产物打透。

2. **规则和维度太多，容易变成硬编码森林**
   - 33 个审计维度、专用词表、反 AI 味规则都很强，但维护成本会快速上升。
   - 我们应该优先保留少量高价值维度，做可观测、可调参，而不是一次性堆满。

3. **CLI 友好，但对面板的交互还可以更强**
   - 他们已经开始做 Web UI，但当前仓库仍然偏 CLI。
   - 对我们来说，`dashboard` 有机会把这些控制面和审计结果直接可视化，形成更好的闭环。

## 对我们项目的具体启发

1. **继续巩固三层结构**
   - 控制面：`author_intent / current_focus / chapter intent`
   - 真相层：`state / story_memory / index / vectors`
   - 运行层：`context / trace / audit / dashboard`

2. **把写作链编译化**
   - 先编译上下文，再写作，再审计，再修订。
   - 这样更容易定位“是记忆错了、上下文错了，还是模型写歪了”。

3. **dashboard 要成为编译结果的展示面**
   - 首页显示本章任务书、召回、记忆健康、风险提示。
   - 记忆页显示可展开的证据，而不是一堆原始列表。

4. **记忆系统要继续向“可裁决”演进**
   - 需要稳定的 Source of Truth、证据链、权威度和回滚/归档策略。
   - 这是比单纯增加召回内容更重要的下一步。

## 建议落点

- 参考其“控制面 / 真相层 / 运行层”结构，补齐我们的术语和文档。
- 参考其 `plan -> compose -> write -> audit -> revise` 链路，继续优化我们的写作任务书和审计回路。
- 参考其 SQLite 记忆检索思路，继续把召回变成编译链的一部分，而不是后置补丁。

## 对补充评估的修正判断

### 认同的部分

1. **Observer / Reflector 分离值得借鉴**
   - 这是比“多 Agent”本身更关键的点。
   - 观察层只抽事实，反思层只产出结构化 delta，再由代码层校验入库，确实比“直接从正文写状态”更稳。

2. **情感弧线是我们当前明显缺口**
   - 这个不是题材特化需求，而是长篇人物稳定性的通用维度。
   - 适合进入 `story_memory` 的独立子层，而不是继续塞进通用 changes。

3. **current_focus 控制面值得优先做**
   - 这个比增加更多审查维度更立竿见影。
   - 它能直接约束“最近几章到底该聚焦什么”，减少写偏。

4. **去 AI 味系统确实是质量差距**
   - 但它不该先做成大词表系统，应该先做成轻量审查维度和疲劳表达告警。

### 需要降温的部分

1. **SQLite 时序记忆不是我们当前 P0**
   - 我们已经有 `index.db + vectors.db + story_memory.json`。
   - 当前主要瓶颈还是记忆裁决、控制面和任务书编排，不是“没有 SQLite”。
   - 所以这个更适合 P2，而不是现在就重做底座。

2. **7 个真相文件不一定优于当前合并模型**
   - 拆文件的好处是清晰，但也会增加同步与维护复杂度。
   - 我们更适合先做逻辑分层和 schema 分区，再决定要不要物理拆文件。

3. **守护进程自动写章不是当前优先项**
   - 这更偏运行模式升级，不是核心写作质量升级。
   - 在控制面、任务书、审计和记忆闭环没完全稳定前，自动化只会放大错误。

4. **同人模式 / 文风仿写有价值，但不该先做**
   - 这是能力扩展，不是底座收敛。
   - 对我们当前主线帮助不如“控制面 + 情感弧线 + 轻量去 AI 味”。

## 调整后的优先级

| 优先级 | 项目 | 判断 |
|--------|------|------|
| P0 | `current_focus` 控制面 | 应尽快做，收益高、改动小 |
| P0 | 情感弧线追踪 | 应尽快做，补齐人物连续性短板 |
| P1 | 轻量去 AI 味审查 | 应做，但先从审查和提示入手，不先做重词表 |
| P1 | Observer / Reflector 分离 | 值得做，但应在 Data Agent 改造时渐进落地 |
| P2 | SQLite 时序记忆增强 | 有意义，但不是当前最痛点 |
| P3 | 文风仿写 / 同人模式 / 守护进程 | 能力扩展，后置 |
