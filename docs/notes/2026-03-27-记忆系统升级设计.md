# 记忆系统升级设计

> 版本：v1.0
> 日期：2026-03-27
> 状态：已批准，待实现

## 背景

200万字小说（≈666章）写作中，长程记忆召回是核心痛点。现有三层记忆架构（state.json / index.db / vectors.db）存在：
- Context 爆炸：前3章摘要撑不到50章
- 隐式信息丢失：情绪节奏等状态没入库就丢失
- 召回粒度粗：只能问"在哪"，无法问"为什么"

## 设计原则

1. **简约而不简单**：功能少但每个都有用
2. **记忆即召回**：不只记录，要能被 LLM 消费
3. **容量硬上限**：防止膨胀，500行上限自动淘汰
4. **触发自动化**：每5章自动 + 写前手动

## 核心设计

### 记忆写入格式

```
PROJECT_ROOT/.webnovel/memory/
└── story_memory.json    # 核心记忆文件（≤500行）
```

**story_memory.json 结构：**

```json
{
  "last_consolidated_chapter": 45,
  "last_consolidated_at": "2026-03-27T10:00:00Z",
  "characters": {
    "萧炎": {
      "current_state": "斗王巅峰，闭关中",
      "last_update_chapter": 45,
      "milestones": [
        {"ch": 10, "event": "突破斗师"},
        {"ch": 25, "event": "获得金帝焚天炎"},
        {"ch": 45, "event": "突破斗王巅峰"}
      ]
    }
  },
  "plot_threads": [
    {"id": "pt_001", "name": "萧炎中毒", "foreshadow_ch": 3, "status": "pending"},
    {"id": "pt_002", "name": "玄铁令", "foreshadow_ch": 8, "status": "pending"}
  ],
  "recent_events": [
    {"ch": 46, "event": "萧炎突破斗王"},
    {"ch": 47, "event": "与纳兰嫣然关系恶化"}
  ]
}
```

### 触发时机

| 触发条件 | 动作 |
|---------|------|
| 每5章写完 | 自动触发记忆整理 |
| `/webnovel-write` 写新章节前 | 自动召回 + 写入新事件 |
| `/webnovel-query` 查询时 | 手动召回 |

### 记忆整理规则（每5章）

**写入条件（只记录转折点）：**
- ✓ 角色境界突破/退步
- ✓ 关系重大转变（结仇/决裂/结拜/爱情）
- ✓ 伏笔埋下（标记 `status: pending`）
- ✓ 伏笔回收（标记 `status: fulfilled`）
- ✓ 关键物品易主/消失
- ✓ 势力格局变化

**不记录：**
- ✗ 日常对话
- ✗ 场景描写
- ✗ 小的状态波动

**淘汰规则：**
- 每个角色 milestones 保留最近 10 条
- recent_events 保留最近 50 条
- plot_threads 只保留 `pending` 状态
- 项目总行数 ≤ 500 行，超出时淘汰最旧记录

### 召回格式

写新章节前，Context Agent 自动加载：

```
## 角色状态快照
- 萧炎：斗王巅峰，闭关中（Ch.45）
- 纳兰嫣然：斗灵，已离开萧家（Ch.40）

## 未回收伏笔
- 萧炎中毒（埋：Ch.3）→ 尚未出现
- 玄铁令（埋：Ch.8）→ 尚未出现

## 近5章重大事件
- Ch.46: 萧炎突破斗王
- Ch.47: 与纳兰嫣然关系恶化

## 上章结尾状态
- 时间：深夜
- 地点：萧家密室
- 情绪：紧张
```

## 实施计划

### Phase 1：基础建设（v5.9.0）
- 新增 `story_memory.json` 存储结构
- 修改 Data Agent，在写完每章后检测是否需要写入记忆
- 每5章触发一次自动整理
- 写 `/webnovel-write` 时自动携带召回的记忆

**测试计划：**
1. 初始化一个有10章的项目
2. 写第11章，验证 story_memory.json 是否正确生成
3. 检查召回内容是否正确

### Phase 2：伏笔追踪（v5.10.0）
- `/webnovel-write` 时，Context Agent 自动标记未回收伏笔
- 写完章节后，检测伏笔是否被回收
- 在写作上下文前增加"伏笔提醒"

**测试计划：**
1. 埋下一个伏笔（如"萧炎中毒"）
2. 写5章后的情节，手动标记为"已埋"
3. 在第6章中回收伏笔
4. 验证 story_memory.json 中状态正确更新为 fulfilled

### Phase 3：容量控制（v5.11.0）
- 实现自动淘汰最旧记录
- 确保 story_memory.json 始终 ≤ 500 行
- 增加容量预警（>400行时提示）

**测试计划：**
1. 写入大量事件，验证超500行时自动淘汰
2. 验证淘汰的是最旧记录，不影响新记录

### Phase 4：LLM 原生召回（v5.12.0）
- RAGAdapter 支持优先检索 story_memory.json
- 实现"近5章事件"的语义召回
- Context Agent 自动将召回结果注入上下文

**测试计划：**
1. 写入特定情节
2. 用 `/webnovel-query "查找最近的冲突事件"`
3. 验证召回结果包含相关事件
