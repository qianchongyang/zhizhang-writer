# Webnovel Dashboard 重构设计

> **目标**：把现有只读 Dashboard 从“数据浏览器”升级为“写作驾驶舱 + 数据二级页”，让首页优先服务写作决策，再保留现有实体、图谱、章节和文件浏览能力作为辅助工具。

## 背景

当前 `/webnovel-dashboard` 主要展示：
- 数据总览
- 设定词典
- 关系图谱
- 章节一览
- 文档浏览
- 追读力分析

这些页面能看见很多数据，但首页并没有把最新的核心能力放到前面：
- `story_memory`
- 写前召回
- 记忆健康
- 归档召回
- 写作建议

结果是仪表盘仍然更像“查表面板”，而不是写作时会真正盯着看的“操作台”。

## 设计目标

1. 首页优先回答“这一章该怎么写、该避什么、该补什么”。
2. 把记忆相关信息集中到单独页面，避免散落在各个数据页里。
3. 保留现有数据浏览能力，作为二级页与排障入口。
4. 继续保持只读，不引入任何修改项目状态的接口。
5. 兼顾桌面端和移动端可读性，避免首页信息过载。

## 设计原则

- **决策优先**：首页先给结论，再给明细。
- **记忆优先**：`story_recall`、记忆健康、归档与伏笔优先展示。
- **数据下沉**：实体表、图谱、章节列表继续保留，但不占首页第一屏。
- **只读安全**：Dashboard 仍然只提供 GET 接口。
- **渐进升级**：先重排信息结构，再补增强页，不一次性重做全部页面。

## 目标信息结构

### 首页：写作驾驶舱

首页只保留写作最相关的信息，按下面顺序排列：

1. 项目状态
   - 当前章
   - 当前卷
   - 总字数
   - 目标进度
   - 当前题材

2. 本章决策卡
   - 本章大纲摘要
   - 本章写作目标
   - 写作建议
   - 风险提醒

3. 记忆卡
   - 高优先级召回
   - 关键人物状态
   - 未回收伏笔
   - 最近重大事件
   - 归档召回摘要

4. 健康卡
   - 记忆是否滞后
   - 召回是否过少
   - 是否存在伏笔积压
   - 是否存在状态冲突

5. 快捷入口
   - 去写当前章
   - 打开记忆与召回页
   - 打开设定词典
   - 打开关系图谱
   - 打开章节一览

### 二级页：记忆与召回

新增一个独立页面，集中展示：
- `story_memory` 的结构化内容
- `story_recall`
- `archive`
- `recall_policy`
- `memory_health`
- 结构化变化账本

这个页面的目标不是“让用户看得更多”，而是“让用户看得更准”。

### 二级页：保留现有数据页

以下页面继续保留：
- 设定词典
- 关系图谱
- 章节一览
- 文档浏览
- 追读力

它们继续作为查询与排障入口，不作为首页核心。

## 页面布局方案

### 方案推荐：双层模式

#### 第一层：写作驾驶舱
- 顶部左侧显示项目标题与当前章
- 顶部右侧显示同步状态与数据刷新状态
- 主区域采用两栏或三栏布局
  - 左栏：本章大纲与写作目标
  - 中栏：高优先级召回与关键记忆
  - 右栏：记忆健康与风险提示

#### 第二层：数据与分析页
- 使用现有侧边栏导航
- 保留实体、图谱、章节、文件、追读力
- 新增“记忆与召回”页

### 建议新增的数据整合接口

为了避免前端自己拼很多块数据，建议后端补一个只读聚合接口：
- `/api/dashboard/summary`

这个接口返回：
- 项目状态
- 本章大纲
- `story_recall`
- `memory_health`
- `writing_guidance`
- `archive_recall`

这样首页只需请求一个聚合接口，数据结构更稳定，也更容易演进。

## 数据展示规则

### 首页显示规则

- 首页只显示 Top N 信息，避免过长。
- 伏笔只显示最相关的 5 条。
- 召回只显示最相关的 5 条。
- 记忆健康只显示最关键的 4 个指标。
- 其余内容下沉到二级页。

### 颜色与状态

- `green`：正常、稳定、无风险
- `amber`：需关注、存在轻度风险
- `red`：需要处理、明显异常
- `blue`：信息类、一般状态

### 文案要求

- 少讲字段，多讲结论。
- 少讲“数据有多少”，多讲“你现在该做什么”。
- 允许保留字段名，但要配人话摘要。

## 交互建议

1. 支持手动刷新。
2. 支持自动刷新提示，但不要频繁闪烁。
3. 卡片支持展开/收起，默认收起次要明细。
4. 章节切换后，驾驶舱应自动刷新当前章节对应信息。
5. 记忆页支持按 `memory_tier` 筛选。

## 容错与降级

- 如果 `story_memory` 不存在，首页仍然显示项目状态与现有数据页。
- 如果聚合接口失败，首页退回现有分散接口。
- 如果某个数据块为空，显示明确的空状态，不显示假数据。
- 如果 `story_memory` 损坏，记忆页显示降级说明，不阻塞整个 Dashboard。

## 测试建议

### 前端测试

- 首页能正常显示写作驾驶舱模块。
- 记忆页能正确渲染 `story_recall` 和 `archive_recall`。
- 空数据情况下不崩溃。

### 后端测试

- 聚合接口返回完整结构。
- 缺失 `story_memory` 时能降级。
- 路径安全校验不被破坏。

### 回归测试

- 旧的实体页、图谱页、章节页、文件页和追读力页仍可访问。
- 现有只读接口不受影响。

## 分阶段实施建议

### Phase 1：重排首页
- 先把首页从“数据总览”改成“写作驾驶舱”
- 不动现有二级页结构

### Phase 2：新增记忆与召回页
- 集中展示 `story_memory`、`archive`、`memory_health`
- 把召回链路可视化

### Phase 3：补聚合接口
- 增加 `/api/dashboard/summary`
- 让首页减少多次请求与前端拼装

### Phase 4：微调二级页
- 让现有数据页更像辅助分析，而不是首页主入口

## 验收标准

1. 打开 `/webnovel-dashboard` 后，第一屏能直接看到当前章写作所需信息。
2. 首页能明确展示 `story_recall`、未回收伏笔、关键人物状态和记忆健康。
3. `记忆与召回` 页能展示 `story_memory`、`archive` 与召回政策。
4. 现有数据页仍可使用，且不影响只读特性。
5. 旧项目在没有完整 `story_memory` 时也能正常显示。

## 风险

- 首页重排后，如果信息密度控制不好，仍可能显得拥挤。
- 聚合接口如果做得太早，可能绑定过多前端依赖。
- 记忆页展示过多细节，会削弱首页“决策优先”的目标。

## 结论

Dashboard 最值得做的升级，不是继续加更多表，而是把它变成一个真正的写作决策界面：
- 首页看决策
- 记忆页看召回
- 数据页看明细

这样才能跟当前的 `story_memory`、召回、归档、健康报告能力对齐。
