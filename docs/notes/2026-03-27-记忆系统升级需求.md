### Mem0 四层记忆分类（最适合参考）

```
┌─────────────────────────────────────────────────┐
│ 1. Working Memory（工作记忆）                    │
│    当前会话中的即时状态                            │
│    例：正在写的情节、当前角色的即时情绪            │
├─────────────────────────────────────────────────┤
│ 2. Factual Memory（事实记忆）                    │
│    实体属性、关系、数值——"是什么"               │
│    例：萧炎的境界是斗王巅峰、注注意单有5颗        │
├─────────────────────────────────────────────────┤
│ 3. Episodic Memory（情景记忆）                    │
│    事件序列、时间线——"发生了什么"               │
│    例：Ch.45 萧炎突破斗王、Ch.46 吃了第一颗丹药  │
├─────────────────────────────────────────────────┤
│ 4. Semantic Memory（语义记忆）                  │
│    概念关系、因果推理——"为什么"                 │
│    例：突破斗王需要闭关、丹药需在拍卖行获取      │
└─────────────────────────────────────────────────┘
```

### 记忆系统四维度设计

根据 Martinig Mastery 的七步框架，记忆系统需要回答四个核心问题：

| 问题 | 含义 | 小说场景 |
|------|------|---------|
| **What to Store** | 存什么 | 不是所有对话都存，只存"转折点事件" |
| **How to Store** | 怎么存 | 结构化数值 vs 非结构化文本 |
| **How to Retrieve** | 怎么召回 | 语义检索 vs 精确查询 |
| **When to Forget** | 何时遗忘 | 超容量时淘汰最旧记录 |

### 数值一致性：核心难题

你提到的"金钱换算"问题，在 AI Agent 领域是**数值状态追踪**问题：

```
Problem:
  主角有 100下品灵石 + 30上品灵石
  拍卖花掉 20上品灵石
  剩余应该是：100下品 + 10上品

Solution（不能只靠 LLM 推断）：
  1. 数值必须有"精确记录"作为 Source of Truth
  2. 所有变更必须经过"事务性更新"
  3. 召回时必须返回精确数值，不是"大概"
```

**这就是为什么 PostgreSQL 比 SQLite 更适合**：
- PostgreSQL 支持事务（ACID）
- 可以用 `pgvector` 做语义检索
- 数值字段有强类型约束
- 可以建立关系约束（外键、触发器）

## 现状分析

### 三层记忆架构

| 层次 | 存储 | 能力 | 局限 |
|------|------|------|------|
| 运行时状态 | state.json | 主角/关系/势力/时间线/伏笔追踪 | 只存当前状态，历史演化分散 |
| 实体索引 | index.db (SQLite) | 角色/地点/物品/关系网络/别名消歧 | 无法做情节层面的多跳推理 |
| 语义检索 | vectors.db | scene chunk 的向量+BM25混合检索 | 只限"像不像"，不懂"因果" |

### 百章后的痛点

| 痛点 | 现象 |
|------|------|
| Context 爆炸 | 前3章摘要撑不到50章，关系网复杂后上下文窗口溢出 |
| 隐式信息丢失 | 角色的"情绪节奏""喘息未定"等状态没入库就丢失 |
| 召回粒度粗 | 只能问"萧炎在哪"，无法问"萧炎30-50章情绪变化" |
| **数值不一致** | 金钱/物品变化靠 LLM 推断，不可靠 |
| **记忆碎片化** | 没有统一的"记忆定义"，什么该记什么不该记 |
| **更新机制缺失** | 伏笔回收后，原记录怎么更新？没有规范 |

## 什么是小说写作的"记忆"？

### 记忆分类框架

基于 Mem0 和 Memory OS 的研究，我定义小说写作的记忆分类：

| 记忆类型 | 内容 | 例子 | 存储方式 |
|---------|------|------|---------|
| **实体记忆** | 角色/物品/地点的属性 | 萧炎的境界、丹药的品级 | 结构化数值 |
| **关系记忆** | 实体间的关系 | 萧炎-纳兰嫣然的仇敌关系 | 图数据库 |
| **事件记忆** | 发生了什么 | Ch.45突破斗王、Ch.46吃丹药 | 时序事件 |
| **数值记忆** | 可量化的数值变化 | 灵石数量、丹药颗数 | 精确数值 |
| **语义记忆** | 因果推理 | 为什么需要丹药、突破的条件 | 知识图谱 |
| **情景记忆** | 场景氛围 | 萧家密室的紧张氛围 | 文本摘要 |

### 什么需要记忆？

**有效记忆（必须记录）：**
- ✓ 数值变化（灵石、花费、物品数量）
- ✓ 境界/能力变化（突破、退步、受伤）
- ✓ 关系变化（结仇、决裂、结拜、爱情）
- ✓ 伏笔埋下/回收
- ✓ 关键物品获取/消耗
- ✓ 时间线跳跃（闪回、跳到X天后）

**无效记忆（不需要记录）：**
- ✗ 日常对话
- ✗ 场景描写（除非是关键场景）
- ✗ 小的状态波动（今天吃了什么）
- ✗ 重复的情节模式

### 什么需要精确记录（不能用 LLM 推断）？

| 类型 | 为什么必须精确 | 示例 |
|------|----------------|------|
| **货币** | 换算关系复杂，容易算错 | 100铜币=1银币=0.01金币 |
| **物品数量** | 消耗后必须减少，不能靠感觉 | 5颗丹药，吃1颗剩4颗 |
| **境界等级** | 修炼体系严格，不能跳跃 | 斗者→斗师→大斗师 |
| **时间约束** | 倒计时类情节必须精确 | 3天后宗门大比 |

## 系统性设计方案

### 四步设计框架

| 步骤 | 问题 | 答案 |
|------|------|------|
| **1. 存什么** | 哪些信息构成记忆？ | 实体记忆 + 事件记忆 + 数值记忆 |
| **2. 怎么存** | 数据结构设计？ | PostgreSQL（数值）+ 图数据库（关系）|
| **3. 怎么召** | 召回机制？ | 语义检索 + 精确查询 + 时序推理 |
| **4. 怎么更** | 更新机制？ | 事务性更新 + 淘汰最旧记录 |

### 数据库选型建议

| 需求 | 推荐方案 | 原因 |
|------|---------|------|
| 数值精确性 | PostgreSQL | 事务支持、强类型、pgvector |
| 关系网络 | PostgreSQL + pgvector | 图扩展处理多跳关系 |
| 语义检索 | pgvector | 向量+结构化统一存储 |
| 简单场景 | SQLite | 零依赖，但数值一致性需应用层保证 |

**最终推荐：PostgreSQL + pgvector**

### 触发时机设计

| 触发条件 | 动作 |
|---------|------|
| 每5章写完 | 自动触发记忆整理（事件记忆 + 数值更新）|
| 数值变化时 | **实时事务性更新**（物品获取/消耗）|
| 关系变化时 | **实时更新**（结仇/结拜）|
| 伏笔埋下/回收 | **实时标记** |
| `/webnovel-write` 写新章节前 | 自动召回相关记忆 |

## OpenClaw 记忆系统调研结果

### 1. OpenClaw 原生记忆

| 组件 | 作用 |
|------|------|
| `MEMORY.md` | curated 长期记忆，人工维护 |
| `memory/YYYY-MM-DD.md` | 每日日志，append-only |
| `memory_search` | 语义召回（向量搜索） |
| `memory_get` | 针对性读取指定文件/行范围 |

**设计原则**：文件即真理，模型只"记住"写入磁盘的内容。

---

### 2. ZenSystemAI/openclaw-memory 工具包

**核心流程：**
```
会话压缩 → LLM 提取事实(JSON) → Qdrant 向量库存储
```

| Skill | 作用 |
|-------|------|
| `memory-store` | 存储事实 + embeddings |
| `memory-query` | 语义搜索长期知识 |
| `memory-delete` | 按ID删除（GDPR合规） |
| `docs-query` | 搜索嵌入式文档 |

**关键洞察**：补充而非替代 OpenClaw 原生记忆
- `memory_search` → 近期会话上下文
- `memory-query` → 长期事实 + 客户知识

---

### 3. tomdebres/openclaw-memory-stack 四层架构

```
┌─────────────────────────────────────────┐
│            AI Agent                      │
└─────────────────┬───────────────────────┘
                  ▼
┌─────────────────────────────────────────┐
│         agent-memory / helpers           │
└─────────────────┬───────────────────────┘
    ┌─────────────┼─────────────┬─────────────┐
    ▼             ▼             ▼             ▼
┌────────┐  ┌────────────┐  ┌────────────┐  ┌────────────┐
│Local   │  │Document   │  │Conversation│  │Health/     │
│typed   │  │memory     │  │memory      │  │drift/      │
│memory  │  │(pgvector) │  │(raw+emb+   │  │coverage    │
│(JSON)  │  │           │  │summary)    │  │            │
└────────┘  └────────────┘  └────────────┘  └────────────┘
```

| 层级 | 存储 | 用途 | 失效模式 |
|------|------|------|---------|
| Local typed memory | JSON/JSONL | 不需要语义搜索的固定事实 | 语义搜索无法召回 |
| Document memory | pgvector | 频繁变化的文档，可发现 | 不精确 |
| Conversation memory | raw + embed + summary | Agent/用户对话 | 优先级漂移 |
| Health/drift | 健康检查 | 检测记忆系统失效 | 静默失败 |

**核心洞察**：分层解决不同问题，失效模式也不同。强行合并成单一存储会降低信任度。

---

### 4. 可借鉴的设计

| 借鉴点 | OpenClaw 方案 | 我们的现状 | 可改进 |
|--------|--------------|-----------|--------|
| 近期 vs 长期分离 | `memory_search`(近期) vs `memory-query`(长期) | 统一 vectors.db | 分离近期/长期召回策略 |
| 会话→事实提取 | `memory-consolidate.sh` LLM 提取 | 无 | 新增"情节摘要→结构化事件" pipeline |
| 健康检查 | 4层分离 + drift 检测 | 无 | 增加记忆系统健康报告 |
| 分层设计 | typed/文档/会话/健康 | 混在 state.json | 重构为分层架构 |

## Claude Code AutoDream 调研（2026-03）

### 核心机制

AutoDream 是 Claude Code 的后台 sub-agent，定期（每24小时/5次以上会话后）自动执行记忆碎片整理。

**类比**：人类 REM 睡眠时的记忆巩固过程。

### 四阶段执行流程

```
Phase 1 - Orientation
  → 扫描 memory 目录，读取 MEMORY.md，构建当前记忆地图

Phase 2 - Signal Gathering
  → 搜索近期会话记录（JSONL），识别高价值模式：
    - 用户纠正、显式保存、反复主题、重要决策

Phase 3 - Consolidation（核心）
  → 实际整理工作：
    - 去重（相同内容的多个笔记 → 合并）
    - 删除过时信息（引用已删除文件、已被取代的旧指令）
    - 转换相对日期为绝对日期
    - 保持 MEMORY.md ≤ 200 行

Phase 4 - 写入
  → 整理后的记忆写回磁盘
```

### 触发方式

| 方式 | 说明 |
|------|------|
| 自动 | 每24小时/5次以上会话后 |
| 手动 | `/dream` 命令（若已上线） |
| 替代 | 对 Claude 说 "dream" / "consolidate my memory files" |

### 与现有 Auto Memory 的关系

| 系统 | 作用 |
|------|------|
| Auto Memory | 会话中捕获学习内容，写入 memory 目录 |
| Auto Dream | 会话间整理、清理、刷新 memory 笔记 |

### 对我们的借鉴价值

| AutoDream 特性 | 我们可以借鉴 |
|----------------|-------------|
| 去重 + 删除过时信息 | 增加"记忆碎片清理"机制，定期消除冗余/过时的伏笔 |
| 相对日期→绝对日期 | 我们的时间线已经有绝对日期，但可以强化"章号锚定" |
| 保持 MEMORY.md ≤ 200行 | 可为我们的 state.json 设置容量上限 |
| 四阶段流程 | 设计我们的"记忆整理"pipeline |
| Phase2 信号收集 | 我们的"情节事件提取"可以参考这个思路 |

### 我们能利用 Claude 原生功能吗？

**可以直接用的：**
- `/dream` 命令 → 触发 AutoDream 整理项目记忆
- `MEMORY.md` → 存放项目级长期记忆（如"这部小说的核心设定"）

**可以利用的点：**
1. **利用 `/dream` 整理我们的记忆文件**（如有 MEMORY.md）
2. **参考 AutoDream 的四阶段流程**设计我们的"章节记忆整理"
3. **利用 Claude Code 的 memory 机制**存储项目级元记忆
